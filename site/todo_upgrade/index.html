    Plan d'Upgrade du Projet Python Starter Template¶

    Ce document liste les améliorations architecturales, de code et de configuration nécessaires pour moderniser le projet vers Python 3.13+ avec les bonnes pratiques 2025. Les tâches sont ordonnées par priorité et dépendances.

    Phase 1 : Configuration et Outils Modernes¶

    1. Remplacer requirements.txt par pyproject.toml ✅
    2. Créer pyproject.toml avec [tool.poetry] ou [build-system] pour gérer les dépendances, versionnement (setuptools-scm), et configurations Ruff/Pytest.
    3. Migrer les dépendances de requirements.txt vers pyproject.toml.

    4. Supprimer requirements.txt.

    5. Supprimer bumpversion.cfg et intégrer versionnement moderne ✅

    6. Configurer setuptools-scm dans pyproject.toml pour le versionnement automatique basé sur Git tags.

    7. Mettre à jour les fichiers qui utilisent bumpversion (application.py, README.md, docker-compose.yml) pour utiliser setuptools-scm.

    8. Mettre à jour la configuration ✅

    9. Remplacer config.ini par des variables d'environnement (via python-dotenv).
    10. Créer un fichier .env.example avec les variables nécessaires (LOG_LEVEL, LOG_FILE_PATH, etc.).

    11. Supprimer config/ et config_samples/.

    12. Ajouter hooks pre-commit ✅

    13. Créer .pre-commit-config.yaml pour exécuter Ruff, black, isort, mypy, etc., avant chaque commit.

    Phase 2 : Architecture et Code¶

    1. Refactoriser l'architecture en couches modulaires ✅
    2. Réorganiser src/app/ en couches : core/, models/, services/, handlers/, db/, repo/.
    3. Implémenter POO avec typage statique (PEP 484/585).
    4. Appliquer patterns SOLID et injection de dépendances.

    5. Créer des exemples de code pour chaque couche (service, modèle, handler).

    6. Améliorer le système de logs ✅

    7. Intégrer logging avec couleurs (rich ou colorama).
    8. Ajouter rotation et archivage des logs.
    9. Configurer via variables d'env (LOG_LEVEL, LOG_FILE_PATH, LOG_TIMEZONE=Europe/Paris).

    10. Écrire des logs dans toutes les méthodes (debug, info, warning, etc.).

    11. Ajouter gestion d'erreurs et exceptions ✅

    12. Créer des exceptions personnalisées.
    13. Logger les erreurs sans masquer les détails.

    Phase 3 : Tests et Qualité¶

    1. Implémenter TDD avec Pytest ✅
    2. Ajouter tests unitaires, d'intégration et E2E.
    3. Configurer pytest-cov pour couverture 85% minimum.

    4. Générer rapports HTML.

    5. Configurer Ruff pour linting et formatage ✅

    6. Intégrer Ruff dans pyproject.toml.
    7. Configurer hooks pre-commit pour Ruff.

    Phase 4 : Déploiement et CI/CD¶

    1. Mettre à jour Docker ✅
    2. Dockerfile : Passer à Python 3.13-slim, multi-étapes, optimisé.
    3. docker-compose.yml : Ajouter health checks, restarts, variables d'env.

    4. Utiliser stacks Portainer pour orchestration.

    5. Ajouter workflows GitHub Actions ✅

    6. Créer .github/workflows/ci-cd.yml pour tests, linting, build Docker, déploiement.

    Phase 5 : Documentation et Finalisation¶

    1. Générer docs avec MkDocs ✅
    2. Créer docs/ avec guide utilisateur, configuration, architecture, releases, API.
    3. Intégrer diagrammes Mermaid/PlantUML.

    4. Déployer sur GitHub Pages.

    5. Mettre à jour README.md ✅

    6. Ajouter guide d'installation détaillé, utilisation, architecture, exemples.

    7. Inclure badges, sections pour CI/CD, etc.

    8. Ajouter exemples de code ✅

    9. Dans src/, créer des fichiers d'exemple pour illustrer les bonnes pratiques (ex. : UserService, UserModel, AuthHandler).

    10. Nettoyer et finaliser ✅

    11. Supprimer scripts/init_env.sh et init_env.ps1 (remplacer par instructions dans README).
    12. Vérifier compatibilité Python 3.13+.

    13. Tester l'ensemble du pipeline CI/CD.

    14. Automatiser CHANGELOG ✅

    15. Intégrer Towncrier pour génération automatique du CHANGELOG à partir de fragments.
    16. Configurer dans pyproject.toml et workflow CI/CD.

    Priorités et Dépendances¶

    • Les phases 1 et 2 peuvent être parallèles.
    • Phase 3 dépend de la refactorisation architecturale.
    • Phase 4 dépend des configurations modernes.
    • Phase 5 dépend de tout le reste.

    Chaque tâche sera validée avec tests et linting avant de passer à la suivante.
